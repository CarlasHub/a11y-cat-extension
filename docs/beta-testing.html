Controlled beta

Beta testing guide

Private beta builds are intended for controlled testing. They are not public production sign-off.

Recommended order: install the extension, run the scan, review confirmed issues first, then review buckets, limitations, and exports.

Install the Beta

Get the beta package: download the latest ZIP asset from GitHub Releases. If no release asset is currently published, request beta access by opening a public issue with label beta-access in GitHub Issues.

  1. Download the latest pre-built beta package from the Releases page or use a local development build.
  2. Open chrome://extensions.
  3. Enable Developer mode.
  4. Use Load unpacked for a local build, or install the downloaded beta package ZIP from Releases.

What to Test

  1. Open a supported Chrome or Chromium desktop page.
  2. Run the deterministic scan from the toolbar panel.
  3. Review Confirmed Issues first.
  4. Review manual/review buckets and scan limitations next.
  5. Use exports for handoff only after checking sensitive snippets and notes.
  6. Verify real AT behaviour separately with VoiceOver, NVDA, or JAWS where required.

Beta readiness matrix

Use this matrix to decide whether a scan result is ready for engineering action or still needs review evidence.

Confirmed Review Advisory Limitation Diagnostic Beta / pending
Beta triage readiness and evidence expectations
Output type Ready for direct defect filing Required evidence Extra validation needed
Confirmed Confirmed issue Yes, when reproducible in current state Issue card details + selector + screenshot Retest after fix
Review Manual/review item No, validate first Context notes + why judgment is needed Manual UX and accessibility review
Advisory Advisory note No, treat as quality signal Guidance relevance to your standards Policy/team decision
Limitation Limitation output No, this is coverage risk Limitation text + diagnostics export Manual testing in missing contexts

What evidence to provide

Context

Page URL and title if shareable, user flow, state reached, expected result, and actual result.

Classification

Whether the issue was confirmed, review-only, advisory, or a documented limitation.

Evidence

Diagnostics export, screenshots, exported evidence bundle, and browser or extension version where available.

What Not to Scan

  • private customer environments you are not authorized to access
  • production systems with confidential content unless your role explicitly allows it
  • authenticated or internal pages that are outside your testing scope
  • third-party sites where automated requests or inspection would violate policy

How to report bugs

Privacy warning: GitHub issues are public. Do not include confidential URLs, selectors, snippets, or raw exports from private environments.

Report beta bugs through the public repository issue tracker.
