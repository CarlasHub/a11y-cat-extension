Internal Chrome technical guide

Internal testing guide

How internal Chrome users should evaluate the extension responsibly and report useful evidence.

Internal testing goal

The internal evaluation should prove that the extension can support real QA and accessibility review workflows without overclaiming production readiness. Feedback should focus on scan trust, UI clarity, export usefulness, documentation accuracy, privacy boundaries, and unsupported contexts.

Recommended internal test path

  1. 1

    Install the current build

    Record the package source, version, browser, and operating system.

  2. 2

    Run representative scans

    Use simple pages, complex app pages, forms, images, headings, language mismatch fixtures, contrast examples, and restricted contexts.

  3. 3

    Check classification quality

    Confirm that confirmed, needs-review, advisory, limitation, and diagnostic outputs remain distinct.

  4. 4

    Test exports responsibly

    Create CSV, JSON, diagnostics, QA report, and state exports only with safe test pages or authorised data.

  5. 5

    Report issues with evidence

    Include steps, page type, browser, diagnostics if safe, expected behaviour, and actual behaviour.

What internal testing feedback should not claim

Do not claim public Chrome Web Store release, WCAG conformance, production certification, or real screen-reader validation unless separate release evidence proves those facts.
