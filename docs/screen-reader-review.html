Internal Chrome technical guide

Screen Reader Review aid

What the screen-reader review support workflow can organise and what it cannot prove.

Review support, not real AT validation

The Screen Reader Review aid is a workflow for organising review evidence. It is not a replacement for testing with real screen readers and does not prove NVDA, JAWS, VoiceOver, TalkBack, Narrator, switch control, magnification, keyboard-only, or user-task behaviour.

What it supports

  • Preparing a reviewer for real assistive technology testing.
  • Organising expected reading order, visible content checks, notes, and evidence.
  • Reviewing likely issues around headings, landmarks, links, forms, names, roles, and interaction flow.
  • Using the closed-by-default Learn How To Use Virtual Screen Reader disclosure for navigation, forms, dialogs, dynamic updates, and headings prompts.
  • Exporting structured review notes where supported.

Controls and status

The in-panel How to use disclosure explains Start, Stop, Prev, Act, Next, Top, End, Audio, Cursor, Pick element, Cancel picking, Add to QA report, virtual log export, QA report export actions, and Guidepup command tools. The status labels are deliberately scoped: Idle means no virtual pass is active, Running means the Guidepup virtual review source is active, Complete means the stopped pass can be exported or added to the QA report, and Unavailable means the virtual bridge or browser capability is not available in that context. If Pick element is armed and the click lands on the A11Y Cat panel instead of the page, the panel reports that diagnostic and keeps picking active so the reviewer can click a page element or cancel. Audio (TTS) is browser speech synthesis playback of Guidepup virtual phrases, not real screen-reader audio; if Chrome accepts playback but no sound is audible, check site audio permission, system volume, output device, and installed OS/browser voices.

Responsible workflow

  1. 1

    Run the page scan

    Collect current-page scanner evidence and diagnostics.

  2. 2

    Open Screen Reader Review

    Use the aid to identify likely reading-order and interaction paths.

  3. 3

    Test with real assistive technology

    Run the target AT/browser combination for the user journey.

  4. 4

    Record differences

    Capture where real AT output differs from the aid or where manual testing finds additional issues.

  5. 5

    Export evidence carefully

    Share only with authorised recipients because exports can include page context and notes.
