Extension UI feature guide

This page is the strongest user-facing reference for the shipped extension UI. It explains each visible feature, count, category, export, and limitation with implementation-backed taxonomy.

Boundary: A11Y Cat supports accessibility review workflows. It does not replace manual testing or real assistive-technology validation.

Quick result-reading guide

Confirmed issues only Critical Serious Moderate Minor Review Manual / Incomplete / Review Visual Composition Review Human Judgement Review Advisory Notes Limitations Diagnostics Local-only state Beta / pending
  1. Start with Confirmed Issues.
  2. Fix Critical and Serious confirmed findings first.
  3. Review Manual / Incomplete / Review items before calling them defects.
  4. Read Scan Limitations before assuming full page coverage.
  5. Treat Advisory Notes as guidance, not automatic failures.
  6. Review export content for sensitive snippets, selectors, and URLs before sharing.

False-positive/false-negative note: automated output can over-report or miss issues. Confirm high-impact findings manually and include real assistive-technology testing for sign-off.

Corroborated Failures

Confirmed findings with additional supporting signals.

What it does

Indicates higher-confidence confirmed findings with corroboration metadata.

Where the data comes from

A11Y Cat corroboration logic and source attribution.

How to read the result

Confirmed.

What the user should do next

Prioritize and remediate as confirmed defects.

Limitations

Corroborated does not mean a human has verified the issue, and it does not prove full compliance coverage.

Example

A corroborated failure is a confirmed issue where more than one signal supports the same finding. For example, axe-core may report an image-related failure and A11Y Cat's custom alt-text analysis may independently identify the same element as lacking meaningful text alternative evidence in the same scan.

Critical workflow diagrams

Previous scan comparison flow

Comparison is scoped to the same origin + pathname. Query strings and hash fragments are ignored for scope matching.

  • Text equivalent: comparison only runs when a previous scan exists for the same origin+pathname scope.
  • Text equivalent: query/hash differences do not force different comparison buckets.

Screen Reader Review limitation flow

Virtual Screen Reader Review supports structured notes and export, but it does not replace real assistive-technology evidence.

  • Text equivalent: virtual SR data can support triage and handoff but cannot certify AT behavior.
  • Text equivalent: manual VoiceOver, NVDA, and JAWS verification remains required.

WCAG and compatibility scope

Implementation scope for WCAG-tagged automated scanning and runtime compatibility.

The scanner runtime uses axe-core with WCAG A/AA tag scope for WCAG 2.0, 2.1, and 2.2 in the current build configuration. The extension reports mapped criteria where available, but it does not prove WCAG conformance.

See Compatibility and WCAG scan scope for the exact runtime tag list and browser/operating-system status.

Glossary

  • axe-core: bundled accessibility rules engine.
  • confirmed issue: deterministic/corroborated finding in current tested state.
  • incomplete: engine could not fully determine result.
  • manual review: requires human judgement.
  • advisory: quality guidance, not automatic failure.
  • deterministic failure: confirmed deterministic issue class.
  • corroborated failure: confirmed issue with additional supporting evidence.
  • visual composition review: visual judgement required.
  • human judgement review: context interpretation required.
  • engine-limited review: runtime/engine could not safely conclude.
  • state-limited review: current state only inspected.
  • scan limitation: explicit coverage boundary signal.
  • source engine: origin of finding logic (axe or custom module).
  • WCAG mapping: criterion references attached where available.
  • selector: DOM locator for evidence target.
  • evidence snippet: short extracted context from inspected page.
  • local history: locally retained scan snapshots.
  • previous scan: newest earlier saved scan for same origin+pathname scope.
  • export: user-triggered local evidence output file/copy.
  • diagnostics: structured runtime support signals.
  • unsupported context: runtime state where full inspection is unavailable.
  • restricted page: page blocked by browser/origin policy constraints.
  • virtual screen reader review: simulated review aid, not real AT runtime proof.
