UI Feature Guide

Use this page when you need the real tab-by-tab and control-by-control guide. It explains what to click, which local libraries or browser APIs power each surface, why the feature matters for accessibility, and where human review still has to take over.

How to use this guide

  1. 1

    Start with the tab or control you are actually using

    Each section below matches a visible extension surface, not an internal code module. That keeps the guide useful for non-technical reviewers first.

  2. 2

    Use the technology rows when you need implementation context

    Every feature section names the shipped libraries or Chrome APIs behind that surface so engineers and release reviewers can trace the public claim back to the runtime model.

  3. 3

    Keep the evidence boundary in view

    Confirmed issues, needs-review items, advisory results, limitations, and diagnostics do not mean the same thing. Use the classification table and the limitation notes before turning any item into a defect or release claim.

Scan Results

Scan Results is the main triage surface. It is where reviewers see confirmed issues, needs-review items, severity counts, issue details, highlight actions, and export entry points after a page scan finishes.

  1. 1

    Open the extension on the page you want to inspect

    The user action triggers the Manifest V3 service worker and temporary activeTab access for that page state.

  2. 2

    Run the scan and review grouped results

    The panel separates confirmed issues from review, advisory, limitation, and diagnostic evidence so the first screen does not flatten everything into the same severity.

  3. 3

    Use filters and issue details

    Reviewers can narrow by classification, severity, source, and search terms, then open item detail, selector, snippet, and provenance information.

  4. 4

    Choose the next action

    Use highlight, copy selector, workflow state, export, or handoff actions where supported, then confirm important items in the real user journey.

AspectDetail
Powered bychrome.action, activeTab, chrome.scripting, bundled axe-core, and A11Y Cat result-shaping modules.
Accessibility relevanceThis is the main entry point for finding high-confidence implementation problems and separating them from items that still need manual judgement.
Evidence valueStrongest page-state evidence in the extension, plus the first place users see severity, provenance, review state, and export options together.
Manual boundaryThe surface still represents only the current rendered state. Hidden routes, delayed states, authenticated paths, and assistive-technology behaviour still need separate review.

Confirmed contrast evidence also appears in Scan Results when the runtime has strong enough proof. The former Contrast Inspector tab is not exposed in the active extension, so uncertain contrast cases remain review or diagnostic evidence instead of being overstated as confirmed failures.

Metadata Check

Metadata Check reviews document-level information such as title, description, canonical URL, Open Graph, Twitter card, and structured data. It helps reviewers catch page-quality and orientation problems that matter to users and downstream sharing surfaces.

  1. 1

    Open Metadata Check after the scan

    Use it when you want page-level naming and preview information, not just DOM-level accessibility findings.

  2. 2

    Review missing, malformed, or conflicting fields

    Look for missing titles or descriptions, broken canonical values, weak preview metadata, or mismatches between page title and the visible H1.

  3. 3

    Decide whether the output is accessibility-critical or quality guidance

    Metadata can affect orientation, findability, and assistive context, but it is often advisory rather than a direct WCAG conformance result.

AspectDetail
Powered byA11Y Cat metadata and structured-data parsing modules reading document metadata, preview fields, canonical data, and schema signals locally in the browser.
Accessibility relevancePage titles, descriptions, and label consistency help orientation, tab identification, and review readiness even when the output is not framed as a direct accessibility defect.
Evidence valueMainly advisory or diagnostic evidence that can still be useful in QA handoff or documentation review.
Manual boundaryProject policy may treat metadata issues differently. Missing metadata is not automatically proof of a WCAG failure.

Headings

The Headings surface helps reviewers inspect semantic structure, visible H1 use, skipped levels, ARIA headings, hidden headings, and heading-like visual text that does not map cleanly to markup.

  1. 1

    Open Headings when structure and navigation matter

    Use it for long pages, dashboards, documentation, and flows where screen-reader or keyboard users depend on a sensible outline.

  2. 2

    Review the heading order and visible hierarchy together

    Check whether the semantic heading order supports the page content and whether the visible hierarchy suggests a conflicting structure.

  3. 3

    Confirm whether the issue is structural or contextual

    Some pages intentionally break a simplistic outline pattern, so reviewers still need context before treating every heading warning as a defect.

AspectDetail
Powered byA11Y Cat heading-analysis modules inspecting rendered heading elements, ARIA heading roles, and document order.
Accessibility relevanceHeadings support page navigation, content scannability, reading order expectations, and the review work that later happens in real screen readers.
Evidence valueConfirmed, needs-review, or advisory output depending on how strong the structural signal is.
Manual boundaryVisual hierarchy, component semantics, and authoring intent still need human review. Heading output does not replace a full content-strategy or IA review.

Alt Text Analysis

Alt Text Analysis separates missing or clearly broken image alternatives from context-dependent cases such as decorative images, linked images, SVG or icon patterns, complex graphics, and background-image situations that need a reviewer to interpret purpose and context.

  1. 1

    Open Alt Text Analysis when the page uses meaningful imagery

    This surface is most useful on content-heavy pages, product cards, dashboards, and marketing pages where images carry information.

  2. 2

    Check the image role and context

    Review whether the image is informative, functional, decorative, redundant, or complex before deciding what a good alternative should be.

  3. 3

    Use the classification, then validate with content context

    Confirmed missing alternatives are useful remediation candidates, but quality and usefulness of alt text still need content judgement.

AspectDetail
Powered byA11Y Cat image and accessible-name analysis modules reading image elements, alt attributes, accessible names, linked-image context, and nearby cues where available.
Accessibility relevanceText alternatives affect screen-reader output, link purpose, comprehension, and task completion for non-visual users.
Evidence valueCan produce confirmed or needs-review evidence depending on whether the image purpose is obvious from the markup and context.
Manual boundaryDecorative intent, complex images, diagrams, and the real usefulness of alternative text remain contextual and need human review.

Text, language, and link review surfaces

This part of the UI covers text-quality and navigation-supporting checks that are useful in accessibility review but need careful interpretation because they can be language-, project-, or environment-dependent.

Language Mismatch

  1. 1

    Review the declared language and visible text together

    Use the output to spot likely mismatches between the page language metadata and the language detected from visible content.

  2. 2

    Check mixed-language or short-copy cases manually

    Short labels, brand names, user-generated content, or intentionally multilingual pages can trigger review-only evidence.

AspectDetail
Powered byA11Y Cat language-detection heuristics reading lang attributes and visible text samples locally in the browser.
Accessibility relevanceLanguage metadata affects pronunciation, reading rules, and assistive-technology interpretation.
Manual boundaryMixed-language, short, or brand-heavy text still needs human confirmation.

View current Language Mismatch screenshot.

Spelling Review

Where the current build exposes spelling review, it uses the same visible-text pipeline plus a separate local spelling engine and allowlist workflow.

  1. 1

    Review flagged terms in context

    Check the original sentence, nearby UI copy, and the target audience before deciding whether a flagged term is actually problematic.

  2. 2

    Separate editorial issues from approved product vocabulary

    Brand names, technical words, abbreviations, and internal terminology may need an allowlist or editorial decision rather than a defect.

  3. 3

    Record the outcome as advisory guidance

    Use spelling output to improve clarity and trust, but do not overstate it as a direct accessibility conformance failure.

AspectDetail
Powered byA CSpell-derived local engine and dictionaries; no remote spellcheck API is used in the documented workflow.
Accessibility relevanceSpelling affects comprehension, trust, and support-readiness, but it is usually advisory rather than a direct accessibility failure.
Manual boundaryDomain terms, product names, and brand vocabulary may require an allowlist decision or editorial review.

Broken Links

  1. 1

    Open Broken Links when navigation reliability matters

    Use it to review same-origin links and supported request-status evidence from the browser context.

  2. 2

    Check whether the link evidence is complete

    Cross-origin, authenticated, blocked, timeout, or rate-limited states may produce limitations or incomplete evidence instead of a confirmed broken-link result.

AspectDetail
Powered byA11Y Cat link-check modules using browser-context link discovery and same-origin status checks where safe and supported.
Accessibility relevanceBroken navigation can block task completion and create dead ends for keyboard and assistive-technology users.
Evidence valueUsually advisory, needs-review, or limitation evidence rather than a universal WCAG claim.
Manual boundaryCross-origin, authenticated, blocked, or environment-specific links may need manual confirmation outside the extension.

Browser Preference QA

Browser Preference QA helps reviewers simulate or inspect accessibility-related presentation scenarios such as zoom, text spacing, font size, reduced motion, colour scheme, forced colours, and contrast-preference conditions.

  1. 1

    Choose the scenario that matches the review question

    Use this surface when you need to inspect how a page responds to readable-text and reduced-motion conditions, or when you need QA evidence for browser-preference behaviour.

  2. 2

    Apply the preview or inspection utility

    The tool can use extension-side preview state, computed style inspection, reduced-motion inventory, or debugger-backed emulation where supported.

  3. 3

    Review the page response and document the outcome

    Check clipping, overlap, hidden content, motion persistence, forced-colour readability, and whether the issue belongs in a manual QA report or a remediation ticket.

AspectDetail
Powered bychrome.debugger where supported, extension-side preview state, computed style inspection, and reduced-motion inventory signals.
Accessibility relevanceThis surface helps reviewers test readability and motion-reduction expectations that matter to low-vision users, cognitive users, and users with vestibular sensitivities.
Evidence valueManual QA evidence, not a fully automatic pass/fail result.
Manual boundaryPreview styles are review aids, not proof that a browser-level preference or real user setup behaves identically in production.

Screen Reader Review aid

Screen Reader Review is a structured review aid, not a replacement for NVDA, JAWS, VoiceOver, TalkBack, Narrator, keyboard-only testing, or real user-task testing.

  1. 1

    Open the aid after you understand the page structure

    Use it once the main scan and heading or naming evidence have given you enough context to plan a real assistive-technology check.

  2. 2

    Review expected reading order and capture notes

    Use the guided surface to organise what you expect to hear or reach in the user journey, then record observations and next steps.

  3. 3

    Compare with real AT behaviour

    Run the target browser and assistive-technology combination for the release decision and treat differences as higher-priority evidence than the review aid.

AspectDetail
Powered byGuidepup virtual screen reader support, browser TTS where available, and local note-taking or export flows.
Accessibility relevanceHelps reviewers plan non-visual reading-order and interaction checks before or alongside real AT testing.
Evidence valueStructured review preparation and review-note evidence.
Manual boundaryIt never proves real AT output. Real browser/AT combinations still decide the release claim.

History, exports, diagnostics, and support workflows

These features support traceability, handoff, and support. They matter because accessibility review is usually iterative: you need to compare scans, preserve notes, export authorised evidence, and explain coverage limits without overstating what the tool proved.

Previous Scan and History

  1. 1

    Compare like-for-like states

    Use the same route, auth state, viewport, and interaction state when comparing two scans.

  2. 2

    Read changed counts with limitations in mind

    Changes can reflect product changes, but they can also reflect different page state, coverage, or runtime conditions.

AspectDetail
Powered bychrome.storage.local and local history or workflow-state modules.
Accessibility relevanceSupports remediation follow-up and helps teams avoid losing context between scans.
Manual boundaryHistory is not automatic regression proof. Reviewers still need to inspect the underlying evidence and rerun manual checks.

View previous-scan comparison screenshot.

Exports and local state

  1. 1

    Choose the export type that fits the audience

    CSV helps spreadsheet or ticket triage, JSON preserves structured evidence, QA report HTML supports human-readable handoff, and diagnostics or review-state exports support reproduction.

  2. 2

    Confirm the data is authorised to leave the browser

    URLs, selectors, snippets, diagnostics, screenshots, notes, and workflow state can all be sensitive after export.

  3. 3

    Clear local data when the review window ends

    Use clear-data controls and your organisation's retention policy together.

AspectDetail
Powered byLocal CSV, JSON, diagnostics, screen-reader-review, and HTML report builders plus chrome.storage.local for retained state.
Accessibility relevanceExports make accessibility evidence reusable across QA, engineering, accessibility specialists, and support teams.
Manual boundaryOnce a file is exported, A11Y Cat no longer controls retention, deletion, or sharing. Treat exports as sensitive artefacts.

Diagnostics and support

  1. 1

    Open diagnostics when something looks incomplete or blocked

    Use the diagnostic surface when scans are unexpectedly quiet, coverage drops, or a page type refuses to scan.

  2. 2

    Use limitations to explain coverage, not defects

    Diagnostics help reviewers say what the tool could not inspect, which is different from claiming the page passed or failed.

AspectDetail
Powered byA11Y Cat diagnostic collectors for permissions, injection, frames, runtime errors, storage support, export support, and coverage boundaries.
Accessibility relevanceGood diagnostics stop teams from overclaiming coverage and help support teams reproduce edge-case scanning behaviour.
Manual boundaryDiagnostics are not page defects. They are operational evidence about the tool state.
